# Version control

> Git workflows that scale. Branching strategies, commit hygiene, the difference between trunk-based development and Gitflow, and why merge vs rebase keeps coming up.

## Git mental model

A Git repository is a DAG of commits. Each commit is:

- A snapshot of the tree.
- Metadata (author, date, message).
- One or more parents.

Branches are just *named pointers* into the DAG. Tags are immutable pointers. Operations like `merge`, `rebase`, `cherry-pick` are graph manipulations.

Internalising this — Git as a graph, not a sequence — clears up almost every confusion.

The canonical reference: [Pro Git book](https://git-scm.com/book) (free, second edition: Chacon & Straub 2014[^pro-git]).

## Branching strategies

### Trunk-based development

Everyone commits (or merges short-lived branches) to `main` daily. Feature flags hide unfinished work.

- Daily integration → no merge hell.
- Requires good test discipline and feature flags.
- The DORA-elite default.

### GitHub Flow

`main` is always deployable. Feature branches off `main`; PR; merge; deploy.

- Simpler than Gitflow.
- Works for continuous-deployment shops.
- The de facto modern default.

### Gitflow

[Driessen 2010](https://nvie.com/posts/a-successful-git-branching-model/)[^gitflow]: long-lived `develop` and `master` branches, plus `feature`, `release`, `hotfix` branches.

- Heavy for typical web work.
- Useful when you ship via versioned releases (libraries, SDKs, mobile apps).
- The author himself later said most teams don't need it.

### Release branching

Long-lived branches per release; cherry-pick fixes from `main` into release branches.

- Used for products with parallel supported versions (databases, OS distributions).

Pick trunk-based or GitHub Flow unless you have a specific reason not to.

## Commit hygiene

A good commit:

- **Does one thing.** "Add login API + refactor 4 unrelated modules" is two commits.
- **Has a clear subject line** (≤ 72 chars, imperative mood: "Add login API endpoint").
- **Has a body** if context matters: what was changed, why, and what changed about *behaviour* (not just files).
- **References issues / tickets** ("Fixes #123") so the trail survives.

Bad commits:

- "wip", "fix", "asdf".
- Mixed concerns ("update README and rewrite half the codebase").
- Generated by accident ("merge from main").

Squash messy WIP into clean commits before opening a PR.

## Merge vs rebase

Two ways to bring `feature` up to date with `main`:

- **Merge** (`git merge main` into `feature`): creates a merge commit; history shows the actual integration.
- **Rebase** (`git rebase main` from `feature`): replays your commits on top of latest `main`; linear history.

Trade-offs:

- Merge preserves true history; merge commits clutter `git log`.
- Rebase produces clean linear history; rewrites your branch (don't rebase shared branches).

Common rule: **rebase your branch on `main` while developing; merge (or squash-merge) into `main` when shipping.** Many teams enforce squash-merge in CI; one PR = one commit on `main`.

## Conventional commits

A loose standard for commit messages:

```
<type>[optional scope]: <description>

[optional body]

[optional footer]
```

Types: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`, `perf`, `ci`, `build`. See [conventionalcommits.org](https://www.conventionalcommits.org/).

Enables automated changelog generation and semver bumping.

## PR / MR culture

A pull request (or merge request) is:

- A branch + a description + a review + a merge.
- The unit of change traceability.
- The artifact your team's history is reconstructible from.

PR description should answer:

- **What** changed.
- **Why** (link to ticket / design doc).
- **How to test** locally.
- **Risks** and rollback plan.
- **Screenshots / before-after** for UI work.

## Tagging and releases

Tags are immutable pointers. Use **semver** ([Preston-Werner 2013](https://semver.org/)[^semver]): `MAJOR.MINOR.PATCH` where:

- MAJOR — breaking API change.
- MINOR — backward-compatible feature.
- PATCH — backward-compatible bug fix.

A release is typically: bump version → tag → CI builds artifact → push tag triggers release pipeline.

## Monorepo vs polyrepo

| | Monorepo | Polyrepo |
|---|---|---|
| Cross-cutting refactor | Easy | Coordinated PRs across repos |
| CI complexity | One pipeline | One per repo |
| Tool maturity | Bazel, Pants, Nx, Turborepo | Trivial |
| Onboarding | Whole company at once | Per-team |
| Storage | Big | Small per repo |

Google, Meta, Twitter, Uber all run monorepos. Most startups start polyrepo. Both work; the choice is operational.

## Hooks and protection

- **Branch protection** on `main`: require PRs, CI passing, ≥1 reviewer.
- **Pre-commit hooks**: format, lint, fast tests.
- **Server-side hooks**: ban large binaries, secrets, force-pushes to main.
- **`.gitignore`** and **`.gitattributes`**: keep noise out, normalise line endings.

## Git tricks worth knowing

- `git bisect` — binary search through commits to find the one that broke something.
- `git reflog` — recover from "I lost my work" with `git reset --hard`.
- `git stash` — temporarily set aside uncommitted changes.
- `git cherry-pick` — apply a single commit from another branch.
- `git rebase -i` — squash, reorder, edit commits before sending PR.
- `git worktree` — multiple branches checked out simultaneously.
- `git log -S 'string'` — find which commit introduced a string ("pickaxe search").
- `git blame -L 10,20 file` — who last touched these lines.

## Big files and data

Git scales badly to large binary files. Solutions:

- **Git LFS** — separate large-file store with Git pointers.
- **DVC** — data version control for datasets and ML models, references in Git.
- **Object storage + manifests** — pure-text Git, with hashes pointing into S3.

For AI engineering specifically: model weights, embeddings, eval datasets all want DVC or S3, not Git directly.

## For AI engineering specifically

- Treat **prompts as code** in Git — see [Prompting → MLOps](../../prompting/prompt-engineering-mlops.md).
- Treat **eval sets as versioned artifacts** — DVC, with a hash referenced from Git.
- Treat **model checkpoints** as versioned with semver and tagged.
- Treat **notebooks** with care — they're code with embedded outputs; use `jupytext` to keep them diff-friendly, or convert to scripts before promoting.
