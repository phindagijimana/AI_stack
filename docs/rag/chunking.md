# Chunking

> Splitting documents into the units the retriever indexes. The decision that quietly bounds RAG quality more than embedding choice.

## Why chunking matters

A retriever returns vectors. Each vector represents one chunk. If your chunks are too large, retrieval is imprecise (everything matches everything). If they're too small, you lose context (the chunk doesn't make sense on its own). The right unit depends on your corpus and your queries.

## Defaults that work

- **Size**: 200–500 tokens (≈ 800–2000 characters).
- **Overlap**: 10–20% of chunk size.
- **Respect** paragraph and heading boundaries when possible.

This recipe gets you to "good enough" on most corpora. Tune from there.

## Token-aware splitting

Character-based splitting (`text[i:i+size]`) is fine for prototypes but slices mid-token, which hurts retrieval. Use a tokenizer:

```python
import tiktoken
enc = tiktoken.encoding_for_model("gpt-4o")

def chunk_tokens(text: str, size: int = 400, overlap: int = 60) -> list[str]:
    ids = enc.encode(text)
    chunks = []
    for start in range(0, len(ids), size - overlap):
        piece = enc.decode(ids[start : start + size])
        chunks.append(piece)
        if start + size >= len(ids):
            break
    return chunks
```

This matches what the LLM (and most embedders) will see at training and inference time.

## Structure-aware splitting

For Markdown, code, or HTML, split on structural boundaries first:

```python
from langchain_text_splitters import MarkdownHeaderTextSplitter

splitter = MarkdownHeaderTextSplitter(
    headers_to_split_on=[("#", "h1"), ("##", "h2"), ("###", "h3")]
)
chunks = splitter.split_text(md_text)
```

Each chunk inherits the heading path as metadata. Two benefits:

- Chunks are semantically coherent.
- The heading path is searchable and tells the LLM where the chunk came from.

For code: split on function / class boundaries. For HTML / web pages: extract main content with `trafilatura` first, then chunk.

## Sliding-window vs hard splits

- **Sliding window with overlap**: every consecutive pair of chunks shares N tokens. Improves recall when the answer straddles a chunk boundary.
- **Hard splits**: no overlap. Simpler; misses some boundary answers.

Overlap is cheap (~20% more chunks); the recall improvement is usually worth it. Set overlap so that any 50-token answer fits entirely inside at least one chunk.

## Contextual chunking [Anthropic, 2024](https://www.anthropic.com/news/contextual-retrieval)[^contextual]

The big retrieval insight of 2024: prepend a 1–2 sentence summary of the document and the chunk's location *to the chunk itself* before embedding. The summary is generated by a cheap LLM.

```
[Document is "Llama 3 model card"; section is "Training data".]
Pretraining used 15T tokens of...
```

Empirically reduces retrieval failures by 30–50% on the original Anthropic eval. Cost: one cheap-model summarisation call per chunk at ingest time. Cacheable via prompt caching for repeat documents.

## Late chunking [Günther et al., 2024](https://doi.org/10.48550/arXiv.2409.04701)[^latechunk]

Instead of embedding each chunk separately:

1. Pass the **whole document** through the embedder (which must support long context).
2. Mean-pool token embeddings *within* each chunk's token range.

Each chunk's embedding incorporates the full document context. Better than independent chunking on coreference-heavy docs (every "the model" gets the right antecedent). Requires a long-context-capable embedder (jina-embeddings-v3, nomic-embed-text-v1.5 with extension).

## Hierarchical chunking

Store chunks at multiple sizes:

- **Sentence-level** (~50 tokens) for precise matches.
- **Paragraph-level** (~200 tokens) for context.
- **Section-level** (~1500 tokens) for synthesis.

Search at the small level, expand to the medium / large for the LLM context. Costs more storage; often big quality win.

## Special structures

- **Tables** — never split a row across chunks. Render as Markdown, keep the column header in every chunk.
- **Code** — split on function/class boundaries; preserve imports.
- **Conversations / chats** — each turn (or thread) is a natural chunk boundary.
- **Slides** — one slide = one chunk; include the deck title as metadata.

## What to put in chunk metadata

At minimum:

- `source` (file path / URL).
- `chunk_index` (position in the document).
- `parent_id` (for hierarchical retrieval — link a small chunk to its big chunk).
- `created_at` / `updated_at`.
- `tags` (custom).

Metadata enables filtering (`source = "runbooks"`), recency boosting (`created_at > 2025-01-01`), and provenance citation (`source = "billing.md, section 'Refunds'"`).

## How to know your chunking is good

A practical test: hand-pick 50 questions from real users. For each, identify the *minimum* set of chunks that should be retrieved to answer it. Measure Recall@10 against that gold set. If Recall@10 < 0.8, fix chunking first; embedding swap won't save you.
